HLAnalysis — Codebase Review & Prioritized Action Plan

Live HL HIP-4 / BTC + Polymarket market-making & hedging engine · multi-agent audit · 2026-05-31
Executive summary. This is a sophisticated, well-instrumented system, but it is currently running real money with several critical gaps between what the operator believes is happening and what the code actually does. The dominant theme is silent divergence: the live engine silently runs a different strategy than the backtest validated (theta knobs dropped), the backtest silently fabricates or overstates PnL (frozen PM strike, no-op slippage, zero HL fees, unsettled hedge legs), and the most dangerous live failures are silent (dead feed with no reconnect/alert, stop-loss that can flip a closing trade into a naked short, no dead-man's-switch). The top risks, in order: (1) blocking REST calls on the single event loop can freeze WS ingest and the stop-loss enforcer exactly when the venue is congested; (2) the stop-loss re-fires a full-size IOC every second with no in-flight guard and no reduce-only on HIP-4, which can oversell a hedged position into a naked short; (3) the ingest loop and recorder die silently with no reconnect and no alert, leaving open positions managed against a frozen feed; (4) the backtest freezes the PM strike when the kline cache is stale and leaves hedge legs unsettled, so the entire calibration corpus may rest on fabricated edge; (5) on the 1GB box a recorder disk-full re-buffer loop OOM-kills the protected live engine. Secondary but material: the daily-loss cap is blind to HIP-4 settlement payouts and fails open on any HL error, no Binance reference-feed staleness watchdog exists (the documented geo-block produces a silently-empty stream), and there is no live hedge-execution path at all despite the system being operated as hedged. Architecturally, money-touching code (leg geometry, exit/topup builders, MarketState, PM loaders) is forked and already drifting, and the live theta config can't even set the bipower/exit-gate knobs the deployment was justified on. Fix the loop-blocking, stop-loss oversell, silent-feed-death, daily-loss-blindness, and backtest-fabrication items before any further tuning or capital increase; most are S/M effort with high confidence.

Live engine safety & crash prevention

3 critical 3 high 1 medium 1 low

These are the failure modes that crash the process, freeze the loop, or let the engine keep trading on dead data — each is a direct path to an unhedged/unmanaged live position.

#1 CRITICAL M · ~½ day conf: high

Blocking REST calls run inside the async event loop, freezing all strategies, stop-losses, and WS ingest

hlanalysis/engine/router.py:193hlanalysis/engine/hl_client.py:202-256hlanalysis/engine/runtime.py:719-721hlanalysis/engine/runtime.py:777-781hlanalysis/engine/runtime.py:919

Problem. Every ExecutionClient call (place, open_orders, clearinghouse_state, user_fills, realized_pnl_since) is a synchronous blocking HTTP/SDK call invoked directly from async coroutines with no asyncio.to_thread. _live_place additionally wraps placement in tenacity with up to 3 attempts and several seconds of exponential backoff. On a single-event-loop engine, one slow/flaky HL or PM REST call parks the shared _ingest_loop (WS market data), the heartbeat, the stop-loss enforcer, and every other slot — exactly when the venue is congested and you most need to react.

Fix. Wrap every blocking ExecutionClient call in await asyncio.to_thread(...) (or give clients true async methods); at minimum offload place, open_orders, clearinghouse_state, user_fills/realized_pnl_since. Bound the tenacity total time so a single placement can't park the loop for seconds.

#2 CRITICAL M · ~½ day conf: high

Ingest loop crash is silent and terminal — no reconnect, no alert, trading continues on frozen data

hlanalysis/engine/runtime.py:587-655hlanalysis/engine/runtime.py:652-655hlanalysis/engine/runtime.py:711-725hlanalysis/recorder/runner.py:35-43

Problem. _ingest_loop catches all exceptions, logs 'adapter ingest crashed', and returns — the only market-data task dies with no reconnect and no signal. Scan loops, the stop-loss enforcer, and reconcile keep running on increasingly stale MarketState. New entries are eventually vetoed by stale_data, but the stop-loss enforcer reads breached_stops against books frozen at the last-seen value, so a real adverse move after the feed dies never triggers the stop and held positions sit unhedged. This maps directly to the live HL Tokyo POP outage pattern. The recorder has the analogous bug: an exception escaping stream() permanently completes that venue's task while the process stays alive (systemd never restarts).

Fix. Wrap stream consumption in a reconnect loop with bounded backoff; publish a FeedDown alert on disconnect and FeedRecovered on resubscribe (mirror the recorder watchdog). On prolonged disconnect, latch all slots halted so nothing trades on frozen state. For the recorder, add_done_callback / asyncio.wait(FIRST_COMPLETED) so a dying adapter trips a clean exit and a HealthEvent.

#3 CRITICAL M · ~½ day conf: high

No dead-man's-switch / liveness alert — engine silence is indistinguishable from healthy

hlanalysis/engine/runtime.py:727-760hlanalysis/engine/runtime.py:579-655hlanalysis/alerts/rules.py:98-215

Problem. The heartbeat loop computes d_events per 30s but only writes it to loguru — it never publishes a bus event or Telegram when d_events==0. There is NO alert kind for 'no events ingested', 'heartbeat missing', or 'process down'. A total feed death produces no StaleDataHalt (which only fires when held positions exist AND a book is received-but-stale). If the feed dies, the ingest task crashes, or the t4g.micro OOMs, the operator gets zero signal while the engine may hold open binary positions against a frozen feed.

Fix. Publish an EngineHeartbeat bus event every interval AND add a server-side dead-man's-switch (systemd timer touchfile or external uptime monitor) alerting when no heartbeat in N intervals. Publish a FeedStale/IngestDown alert when d_events==0 while subscriptions are active.

#7 HIGH M · ~½ day conf: high

MarketState._questions and per-tick scan grow unbounded — OOM on t4g.micro and O(N) scan cost

hlanalysis/engine/market_state.py:52hlanalysis/engine/market_state.py:420-439hlanalysis/engine/scanner.py:212-223hlanalysis/engine/runtime.py:585hlanalysis/engine/runtime.py:739

Problem. _questions is a dict only ever inserted into; mark_question_settled flips settled=True but never deletes. The engine subscribes to many PM series (ETH/NVDA/NBA/...) for recorder ingest, so the universe grows continuously. Scanner.scan iterates all_questions() every 1Hz tick and runs match_question on each; ingest's seen_questions grows too. On a 1G-swap box this is the flagged unbounded-_questions OOM (2026-05-31 live eval), and per-tick O(N) work degrades scan latency. Several side maps grow forever too: slot.pm_settlements, slot.pm_alerted_redemption_qidxs, Scanner._last_logged_state, Scanner._pm_strike_seen, Router._last_exit_ts.

Fix. Evict questions once settled AND past expiry + grace; skip settled questions at the top of the scan loop. Periodically prune the per-slot/per-scanner maps keyed by question_idx on the same horizon. Add a test asserting _questions count drops after settlement+grace and cap total retained questions defensively.

#8 HIGH M · ~½ day conf: high

No reject/error circuit-breaker — a rejection storm hammers the venue every tick

hlanalysis/engine/router.py:206-221hlanalysis/engine/runtime.py:711-725hlanalysis/engine/risk.py:45-192

Problem. On venue rejects (insufficient margin, invalid token id, invalid size) the router logs and publishes OrderRejected but takes no throttling action. The scanner re-fires the same entry on the next 1Hz tick, so a structural rejection (the HYPE-short-eats-margin / HIP-4 buy storm in the code comments) produces unbounded repeated REST calls. There is no per-(question,error-class) reject counter that backs off or halts. This risks rate-limiting/ban and, combined with the blocking-call issue, can wedge the loop while the operator gets no automatic protection.

Fix. Count consecutive rejects per (question_idx, error-class); after a threshold, veto entries on that question for a cooldown (or latch the slot halted on repeated margin/auth rejects) and alert. Reset on a successful fill.

#11 HIGH M · ~½ day conf: medium

Order ACK can be lost between persist and DB update, orphaning an in-flight/filled order with no recovery

hlanalysis/engine/router.py:185-200hlanalysis/engine/state.py:146-158

Problem. _place writes a pending OpenOrder row, calls exec_client.place(), then updates the row from the ACK. An OOM-kill (explicitly in scope) between place() returning a fill and update_order_status leaves the order pending in the DB while it actually filled on the venue, with the Position/Entry never booked — the real position is unhedged/untracked for the whole window. There is no idempotency key beyond cloid, so a retried place after a lost ACK could double-send.

Fix. Make restart reconciliation authoritative for pending rows: resolve each via user_fills/open_orders before scanning, and book the Position from user_fills when a pending order vanished (extend the local-ghost branch to create the Position, not just mark filled). Confirm cloid-based idempotency causes live HL/PM to reject duplicate cloids rather than place twice.

#19 MEDIUM M · ~½ day conf: medium

EventBus has no unsubscribe; halted-slot watchdog stops draining and drops settlement Exits

hlanalysis/engine/event_bus.py:24-49hlanalysis/engine/runtime.py:832hlanalysis/engine/runtime.py:939-953

Problem. EventBus.subscribe appends a Queue with no unsubscribe; each PM slot's _continuous_checks_loop subscription lives for the process lifetime. When a slot is halted (if slot.halted: continue) its loop stops draining pm_bus_sub while events keep being published; the queue fills to maxsize=1024 and drop_when_full silently discards settlement Exits, so the RedemptionTimeout watchdog can miss settlements. Without drop_when_full a full non-draining queue would block publish entirely.

Fix. Add unsubscribe/close and call it when a task exits. Drain pm_bus_sub even while halted (or move settlement capture out of the halt-gated path). Confirm drop_when_full is always set for these subscriptions and add a metric/alert when drops occur.

#38 LOW S · <2h conf: medium

exit_cooldowns.json write is fsync-less and the cooldown map grows unbounded

hlanalysis/engine/router.py:86-95hlanalysis/engine/router.py:69

Problem. _save_cooldowns uses tmp+os.replace (atomic) but never fsyncs, and _last_exit_ts is never pruned (one entry per question_idx ever traded, persisted and reloaded each restart). A partially-flushed tmp under power loss yields truncated JSON the loader discards entirely, silently dropping all cooldowns and allowing immediate re-entry churn after restart.

Fix. Prune _last_exit_ts to entries within the cooldown horizon before persisting; fsync the tmp before os.replace; alert (not silently reset) on load failure.

Hedge & risk correctness

1 critical 6 high 4 medium 1 low

The whole point of the system is hedging and not flipping naked. These items either flip a closing action into new directional risk, defeat the loss cap, or run the live strategy without its risk gates.

#4 CRITICAL M · ~½ day conf: high

Stop-loss re-sends a full-size IOC every second with no in-flight guard — can oversell into a naked short

hlanalysis/engine/runtime.py:850-901hlanalysis/engine/risk.py:60-61hlanalysis/engine/hl_client.py:216

Problem. _continuous_checks_loop calls breached_stops every ~1s and submits a fresh IOC sell for abs(sp.qty) at the current bid. The DB position only updates on fill ACK; if the IOC partial-fills, the ACK is delayed, or the loop ticks again before resolution, the next tick re-fires another full-size IOC. The risk gate short-circuits exits (is_exit returns approved immediately) so there is NO size-vs-held check, and on HIP-4 the client strips reduce_only entirely (hl_client.py:216), so HL receives plain sells. This can oversell a closing position into a naked short on the outcome leg — the worst outcome for a hedging book.

Fix. Track in-flight stop-loss orders per question_idx and suppress re-fire until the prior resolves (filled/cancelled). Re-read held qty and clamp sell size before each resubmit. Enforce an explicit reduce-only semantic for HIP-4 by verifying held qty in the router before submitting a sell.

#9 HIGH M · ~½ day conf: high

Stop/exit IOC is single-shot at top-of-book and walks the book down on a falling market

hlanalysis/engine/runtime.py:865-901hlanalysis/engine/router.py:226-246hlanalysis/engine/risk.py:196-212hlanalysis/strategy/theta_harvester.py:931

Problem. Stop-loss exits place one IOC at the current bid_px for full size; for HL max_slippage_pct defaults to 0 so only the top level is lifted. On a thin HIP-4/PM book the IOC partial-fills (or misses), leaving residual. Closure depends on the 1Hz loop re-detecting the breach and re-firing at the then-current (lower) bid — the 'IOC stop walking down the book' live bug from the 2026-05-31 eval. There is no escalation to sweep within a slippage budget, and in legacy exit mode (v3.1 prod default) the only adverse-move exit is edge_held = held_p - bid - fee, which fires late because held_p and bid fall together while the position bleeds toward zero.

Fix. After a partial/missed stop IOC, retry within a bounded slippage budget that crosses enough levels to clear the residual, with an in-flight guard so only one exit is live. Verify position reaches zero before clearing breach state. For live slots, ensure exit_safety_d>0 (theta) is configured and add a defaulted stop_loss_pct / absolute mark-to-bid drawdown stop for nba_wp (its only adverse exit is the late edge_held gate).

#10 HIGH M · ~½ day conf: medium

Daily-loss gate depends entirely on HL fills, omits settlement PnL, and falls back to a structurally-zero local PnL on any HL error

hlanalysis/engine/runtime.py:918-931hlanalysis/engine/scanner.py:199-209hlanalysis/engine/hl_client.py:401-419

Problem. Both daily-loss checks call realized_pnl_since and, on any exception, fall back to dal.realized_pnl_since — documented as a structural under-estimate (closed positions deleted, settlement PnL not in fills). The fallback is 'continue trading' on a network blip. HIP-4 settlement payouts are not HL fills at all, so the cap never sees the dominant PnL component of the binary strategy. The result: a loss cap defeatable by a transient HL outage flipping to the near-zero DAL value, or by losses realized via settlement. The 10s pnl cache also makes the cap react up to 10s late.

Fix. Incorporate persisted settlement realized PnL into the daily-loss computation. On HL error, fail-safe (latch/halt) rather than trade on a known-understated PnL, or require N consecutive successful reads before re-enabling entries after an outage.

#12 HIGH M · ~½ day conf: high

v5 delta_hedged inherits the stale, unprotected v3 baseline — none of the v3.1-final risk gates plumbed through

hlanalysis/strategy/delta_hedged.py:174-200hlanalysis/strategy/delta_hedged.py:159hlanalysis/strategy/theta_harvester.py:1091-1147

Problem. build_v5_delta_hedged constructs ThetaHarvesterConfig from only original v3 fields and omits every safety knob added since: favorite_threshold, edge_max, min_distance_pct, min_bid_notional_usd, gamma_lambda, exit_safety_d, vol_estimator, lm_threshold, topup_*, fee_model/fee_rate, momentum_mr_*. The hedged variant runs the unprotected v3 baseline (the one that lost -$569 on extreme-edge entries before edge_max). The binary leg can enter trades unhedged v3.1 would veto, and binary_delta then leverages the bad entry into a 2-legged position. binary_delta also uses question.strike directly, which is NaN/0 for PM up/down and bucket markets, silently producing a garbage hedge.

Fix. Route v5's binary config through the same builder as build_v3_theta_harvester so all current gates are inherited by construction. Guard binary_delta against non-finite/zero strike and non-binary klass (return 0 + diagnostic).

#13 HIGH M · ~½ day conf: high

Hedge ledger marks itself fully hedged the instant the IOC intent is emitted, before any fill

hlanalysis/strategy/delta_hedged.py:111-121hlanalysis/strategy/delta_hedged.py:100

Problem. _HedgeState.hedge_qty_btc is set to the full target_delta the moment the hedge IOC intent is emitted, before any fill confirmation. If the hedge IOC partial-fills or fails, the ledger believes it is fully hedged and won't re-fire next tick (gap~=0), leaving silent naked exposure. The strategy never sees realized hedge fills, so there is no feedback loop to correct an under-filled hedge — the same fire-and-forget bug as the primary leg, applied to the hedge.

Fix. Update hedge_qty_btc from confirmed hedge fills via a fill-feedback channel, not from the intended target; treat any hedge IOC shortfall as residual exposure that must re-fire next tick.

#14 HIGH L · multi-day conf: high

No live hedge-execution path exists, yet the system is operated as hedged

hlanalysis/engine/runtime.py:504-575hlanalysis/strategy/delta_hedged.pytests/integration/test_hedge_runner.py:114-211

Problem. Hedging exists ONLY in the backtest runner and the delta_hedged strategy; no Binance execution client is wired into EngineRuntime._build_slot (only HLClient and PMClient). The single hedge test drives the backtest runner with synthetic klines, not the live async runtime. There is no live hedge code, no live test, and NO alert for 'hedge order failed', 'hedge leg unfilled', or 'binary filled but hedge rejected'. If/when a hedge leg is added, a binary fill whose paired Binance hedge is rejected leaves a naked directional position with zero observability.

Fix. If hedging is in-scope for live, add the Binance exec client + per-fill hedge intent path with HedgeFailed/HedgeUnfilled alerts and a naked-drift check (binary qty vs hedge qty beyond tolerance -> alert + halt). If not yet live, document it explicitly and ensure no config implies it. Either way add an integration test for the binary-filled-hedge-rejected case.

#15 HIGH L · multi-day conf: medium

Settlement PnL is computed for alerts but never persisted; two close paths can double-book or lose it

hlanalysis/engine/runtime.py:795-815hlanalysis/engine/router.py:391-416hlanalysis/engine/reconcile.py:163-179hlanalysis/engine/runtime.py:918-923

Problem. Two independent paths close a settled position — the reconcile loop (vanished_positions deletes the Position and publishes a settlement Exit) and the scanner/strategy EXIT -> router._close_settled (re-reads, deletes, publishes its own Exit) — keyed by qidx with no single owner, producing two settlement Exit events with potentially different realized_pnl. Worse, realized settlement PnL is only emitted on the bus/alert, never written to a durable settled-PnL table, so it is lost on restart (the 2026-05-31 eval: fill-table -$122 vs settlement-inclusive +$120 true). The daily-loss cap therefore relies entirely on HL fills which exclude HIP-4 payouts.

Fix. Make settlement close idempotent on a single owner (reconciler OR router) keyed by qidx, and persist a durable settlement record (qidx, ts, realized_pnl) before deleting the Position so true PnL survives restart, feeds the daily-loss gate, and reconciles against venue balance. Add a daily PnL-summary alert and a test asserting durable recording.

#26 MEDIUM S · <2h conf: medium

No halt/escalation on material position-qty drift; engine silently adopts venue qty and keeps trading

hlanalysis/engine/runtime.py:430-454hlanalysis/engine/restart_drift.py:83-93hlanalysis/engine/reconcile.py:180-197

Problem. The qty-mismatch case silently adopts HL's qty and fires a DRIFT alert with no severity escalation or halt when local vs venue qty diverges materially. For a hedging book a silent adoption shifts the strategy's have_position branch and risk caps under the operator without forced acknowledgement; a large adoption (an un-booked fill) should arguably halt. There is also no rate/threshold alert for drift count spiking.

Fix. Add a configurable position-drift USD threshold: |hl_qty - db_qty| * avg_entry > X -> escalate to a halt-class alert (optionally latch the slot halted pending operator clear). Surface a periodic 'drift count in last hour' metric.

#27 MEDIUM M · ~½ day conf: medium

Global inventory cap double-counts in-flight orders and ignores cross-venue exposure

hlanalysis/engine/risk.py:79-84hlanalysis/engine/scanner.py:185-191

Problem. max_total_inventory = live_orders_notional + new_notional + sum(|qty|*avg_entry). live_orders can include pending orders already (partially) reflected as positions once filled, and positions are read per-slot/per-venue only. Aggregate exposure is bounded per-account-slot but NOT across slots/venues, so a supposedly-hedged multi-account book has no true portfolio-level exposure bound and the in-flight term can over- or under-count by fill timing.

Fix. Define inventory from confirmed positions plus genuinely-unacked order notional (exclude orders already booked as positions); add a cross-slot/global exposure aggregate if accounts share capital or net risk.

#28 LOW S · <2h conf: medium

Concurrent-positions / inventory caps evaluated against a stale pre-fill snapshot within a scan tick

hlanalysis/engine/risk.py:86-90hlanalysis/engine/scanner.py:185-191

Problem. positions and live_orders_total_notional are snapshotted once per scan tick and reused across every intent emitted that tick, with no provisional increment between intents. Multiple new entries in a single tick are all checked against the same pre-fill snapshot, so the engine can approve several entries that jointly exceed the concurrent-positions or global-inventory cap.

Fix. Accumulate provisional notional/position-count across intents within a single scan tick (or re-snapshot after each placement) so caps reflect same-tick approvals already routed.

#29 MEDIUM S · <2h conf: medium

Reconcile loop keeps mutating positions/orders even when the slot is restart-blocked

hlanalysis/engine/runtime.py:470-474hlanalysis/engine/runtime.py:762-822

Problem. slot.blocked (from the restart-drift gate) only suppresses the scan loop; _reconcile_loop still adopts/overwrites positions, cancels orphan orders, and publishes settlement Exits while the operator is supposedly holding the slot for inspection. Automatic DB/venue mutation races the operator and can erase the very drift evidence the block was meant to preserve.

Fix. When slot.blocked, run reconcile in detect-only mode (emit drift events, no DB mutation, no cancels) until the operator clears the block.

#30 MEDIUM S · <2h conf: medium

Stale-reconcile pre-trade gate fails open when no reconcile has ever succeeded

hlanalysis/engine/risk.py:119-124hlanalysis/engine/runtime.py:430-431

Problem. check_pre_trade only enforces the stale-reconcile guard when last_reconcile_ns > 0, but last_reconcile_ns is optimistically seeded to startup time regardless of whether a reconcile actually succeeded. If the startup reconcile errored (or the periodic loop keeps crashing), the engine still trades for the first intervals on the false assumption that startup == a clean reconcile.

Fix. Only set last_reconcile_ns after a reconcile completes successfully (including the startup drift run); treat last_reconcile_ns==0 as stale (veto) rather than skip-the-check.

Backtest fidelity (sim-vs-live)

2 critical 3 high 2 medium 2 low

These fabricate or overstate backtest PnL, meaning every tuning decision in MEMORY may rest on numbers that won't survive live — the most insidious class for a calibration-driven strategy.

#5 CRITICAL S · <2h conf: high

PM strike silently FREEZES (returns last cached close) when the kline cache ends before the strike timestamp

hlanalysis/backtest/data/polymarket.py:581-589hlanalysis/backtest/data/polymarket.py:556-589

Problem. _binary_strike resolves the PM Up/Down strike via bisect_right(ts_list, strike_ts_ns)-1 then returns klines[idx]['close'] with NO coverage check. If the cache ends before strike_ts_ns (the documented stale-cache: ended 05-09 at $80,198), bisect returns the final index and the function returns the LAST candle's close as the strike — a frozen, arbitrarily-wrong strike, not 0.0 and not an error. The strike defines the entire binary's winning side, so a frozen strike fabricates edge/PnL for every market past the cache horizon, manufacturing profitable-looking entries that would lose live.

Fix. After the bisect, validate coverage and reject/skip the market when strike_ts_ns exceeds the last kline ts (within one bar-width tolerance). Add a cache-coverage assertion at discover() time so a stale cache fails loudly instead of producing fake PnL.

#6 CRITICAL M · ~½ day conf: high

Open hedge (BTC perp) positions are never settled or marked-to-market at end of data

hlanalysis/backtest/runner/hftbt_runner.py:724-743hlanalysis/backtest/runner/hftbt_runner.py:879-920

Problem. hedge_positions is written on a hedge fill but NEVER read again — the end-of-data settlement block settles only the binary pos, and realized PnL sums result.fills only. A v5_delta_hedged run that holds any hedge residual to expiry records only the OPENING leg of the hedge (~$100k notional) with no offsetting close or mark-to-market, so hedged PnL is wrong by the full unrealized hedge value (hundreds to thousands per market). The backtest literally cannot tell whether the hedge protects or destroys PnL — the entire point of v5. The top-up at line 741 also sets avg_entry=h_fill.price instead of a qty-weighted average.

Fix. At end-of-data, close each open hedge_positions entry against the hedge leg's last book/mark and append a closing Fill, mirroring binary settlement. Fix the top-up avg_entry to a qty-weighted average. Add a test asserting a held-to-expiry hedge produces two fills.

#16 HIGH M · ~½ day conf: high

Configured slippage_bps is a silent no-op on fill price in the hftbacktest path — overstates edge

hlanalysis/backtest/runner/hftbt_runner.py:295-308hlanalysis/backtest/runner/hftbt_runner.py:757-778hlanalysis/backtest/runner/hftbt_runner.py:462-465

Problem. For ENTER/EXIT the runner computes slipped = ask*(1+slippage_bps/1e4) and submits it as the IOC LIMIT price, then records the fill from order.exec_price. hftbacktest fills the IOC against the real book at the touch, so exec_price = best ask regardless of slippage_bps; the term only makes the limit MORE aggressive (can only increase fill probability). So the --slippage-bps knob is inert on execution cost in the main path (it bites only the stop-loss fallback Fill). Live IOC against a thin HL/PM book walks levels and pays spread+impact, so the sim systematically overstates edge for a near-resolution taker strategy.

Fix. Apply slippage as an explicit cost on the recorded exec price (exec_px_effective = exec_px*(1+slippage_bps/1e4) for buys), or rely on real multi-level recorded book depth so hftbacktest walks levels. Add a test that raising slippage_bps strictly reduces PnL.

#17 HIGH S · <2h conf: medium

HL HIP-4 backtests model zero taker fee by default (fee_model=flat, fee_taker=0.0)

hlanalysis/backtest/runner/hftbt_runner.py:104-113hlanalysis/backtest/cli.py:567-575

Problem. RunConfig and the CLI both default fee_taker=0.0 with fee_model='flat', and _binary_fee returns px*qty*0.0 = 0. Unless an operator explicitly passes --fee-taker, every HL HIP-4 backtest charges ZERO trading fee on entries, exits, and stop-loss fills, paid twice on a round-trip live.
