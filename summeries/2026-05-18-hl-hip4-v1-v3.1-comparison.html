HL fee model (post-task feedback)

HL HIP-4 fees are asymmetric: buys = 0 bps; sells = 7 bps taker / 4 bps maker. The backtests above used --fee-taker 0.0 (symmetric), which understates v3.1's true cost when its exit is a marketable sell. v1 is unaffected because its "sells" are synthetic settlement bookings at price = 1.0 (the contract pays out — no exchange order).

Bucket = 0 trades — what we know

v3.1's bucket generalisation (theta_harvester._winning_region, added in commit 43316ff) is in the code path, but on this corpus the strategy never enters. Diagnostics break down as:

ReasonTicksShareMeaning
tte_out_of_window6,47147.5%Scan tick > 12h before expiry — expected, normal scanner behaviour.
no_book7,15052.5%Strategy's per-leg loop found zero legs with a tradeable ask at this tick.
edge / no_favorite00%Never even reached the favorite-gate or edge check.

This is suspicious because direct DuckDB queries against the parquet show real two-sided books for several bucket legs in the 12h window — e.g. Q0 leg #80 has 83k snapshots with tradeable asks in 0.85–0.99 over the last 12h. The strategy's per-tick view is built by the hftbacktest engine from per-leg event arrays; one or more of these is likely:

  • The first #-prefixed leg (e.g. #70, which the diagnostic writer probes as "yes_bid/yes_ask") has empty/one-sided books, masking the others.
  • Other legs (#80) do have books but _book_state returns None because hftbt's depth(asset_no) hasn't yet advanced past the gap.
  • The bucket markets recorded so far ran on a different exchange instance with bid_px = [] for the YES legs (the YES of a middle bucket frequently quotes only the certain-loss side).
Action: Before declaring v3.1-on-bucket a verdict, re-run with --max-markets 1 on the bucket with the cleanest book (Q0/#80 above) and add a no_book diagnostic that lists the per-leg book state. The current diagnostic only logs the first two legs.

Caveats

  • Sample size: 11 daily binary expiries vs 363 on PM (~3% the sample). Any single market swings PnL by $20–$100. Treat all numbers as point estimates with $±30 (v1) and $±50 (v3.1) per-market σ. The Sharpe values are not interpretable below ~100 trades.
  • Depth limitation: HL HIP-4 spreads p50 ≈ 173 bps, p75 ≈ 1,265 bps — far wider than PM. The realistic-cost run partially captures this but cannot reproduce intermittent liquidity gaps that would actually halt orders live.
  • Geographic difference: HL retail flow is concentrated in EMEA/APAC perp users speculating on binaries adjacent to BTC perp; PM is US prediction-market users. This is the most likely source of any persistent edge difference and we can't validate it from 12 days.
  • Fee model: All runs used symmetric fee_taker=0; HL is asymmetric (0 / 7 bps). Net impact ~$1 on this window.
  • Bucket result: 0 trades is a wiring/diagnostic outcome, not a strategy verdict. v3.1's bucket code path is exercised by unit tests on synthetic data; the failure mode is specific to the recorded HL bucket books.

Recommendation

  1. v3.1 binary, default cost: Ship to live shadow against v1 for another ~30 expiries. Decision threshold: v3.1 net PnL > v1 net PnL at the 60% confidence level (~50 markets cumulative).
  2. v3.1 binary, realistic cost: Sized at $100/position the strategy stays comfortably profitable after asymmetric HL fees; no config change needed.
  3. v3.1 bucket: Hold. Reproduce the no_book pattern on a single bucket market, surface per-leg book state in diagnostics, and confirm the issue is wiring (not strategy logic) before any live exposure. Target a follow-up tuning run once 30+ bucket expiries are recorded.
  4. v1: Stays live; production config validated on HL HIP-4 without changes.
