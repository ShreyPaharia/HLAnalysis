TE-1 · Intraday IOC stop/exit stops out positions that win at settlement

Urgent bug strategy area:engine pnl-impact
Problem

Exits (exit_safety_d, exit_stop_loss in v1; exit_edge / exit_safety_d in v31) fire as IOC reduce_only with limit_price = held_book.bid_px. In a fast move the bid evaporates between scan and send, the IOC is rejected ("could not immediately match"), and the next scan re-fires at the new, lower bid — the stop walks down a collapsing book. Worse, the strategy thesis is hold-to-settlement (23/23 settled positions won), so these stops are realizing losses on positions whose terminal payoff was positive.

Evidence
  • v1 #1211 (q 1000121): stop walked to a 0.615 fill vs ~0.90 entry → −$95.27 in one fill; the market then settled as a win (residual +$32.91).
  • v31 #1080 (q 20): churned −$110.65 intraday across 56 fills while v1 held the identical market flat to a +$26.08 settlement win.
  • v31 priceBucket intraday = −$287.50, almost entirely #1080 (−$111) + #1140 (−$161).
Code 
late_resolution.py:585-625   # exit_safety_d + exit_stop_loss: time_in_force="ioc", limit_price=held_book.bid_px
theta_harvester.py:850-938   # exit_safety_d + exit_edge, same IOC pattern
Proposed direction
  1. Make the exit marketable with a bounded slippage budget (limit a few ticks through the bid, or IOC-to-depth with a max-slip cap) instead of a naked IOC pinned to a vanishing top-of-book bid.
  2. Gate the intraday stop behind a settlement-direction check: if the held leg is still the favored outcome by the model, widen/suppress the stop rather than dumping into a transient air-pocket.
  3. Materially widen exit_safety_d or add a re-entry lockout so a single whipsaw can't round-trip 5×.
Needs a design pass + backtest, not a one-line fix — pairs with the mid-hold work in mid_hold_tte_stack_2026_05_21.
Acceptance
  • Backtest over the 05-19→05-31 corpus shows reduced intraday exit losses without degrading the 23/23 settlement-win capture.
  • No exit fill more than N ticks through the bid at send time (slippage cap enforced).

TE-2 · Settlement PnL is never persisted → daily-loss halt can fire on a profitable book

High bug risk area:engine
Problem

_close_settled deletes the position and publishes an Exit carrying the settlement realized PnL, but never writes a Fill/position row, so the local DB never records settlement gains. The daily-loss gate reads HL's API first (correct) but falls back to the local DAL on any HL-API failure — and that fallback sees only intraday losses, never settlement wins. A transient HL outage during a down intraday window can therefore spuriously halt a strategy that is net-positive after settlement. (HL's realized_pnl_since was in fact failing continuously during the 05-30 08:35 outage.)

Code 
router.py:391-416    # _close_settled: delete_position + publish Exit(realized) — NO append_fill
state.py:254-274     # realized_pnl_since = sum(fill.closed_pnl) + sum(open position.realized_pnl) → excludes settled
runtime.py:919-922   # primary HLClient.realized_pnl_since; except: fall back to slot.dal.realized_pnl_since
Proposed direction
  • In _close_settled, persist the settlement realized PnL (a synthetic Fill row, or a settlement ledger row) so StateDAL.realized_pnl_since is complete.
  • Make the daily-loss fallback fail-open (skip the cap when the authoritative venue number is unavailable) rather than fail-halt on an under-counted local number.
Acceptance
  • After a settlement, dal.realized_pnl_since(window_start) matches the published Exit PnL.
  • Simulated HL-API failure with intraday loss > cap but positive net does not trigger DailyLossHalt.

TE-3 · Settled questions are never evicted from memory → OOM

High bug area:engine reliability
Problem

MarketState._questions marks markets settled=True on settlement but never removes them; the scanner walks the full dict every tick. The universe grows monotonically (heartbeat questions 4 → 201+ over the window; seen_question table at 3160 rows). On the t4g.micro (384 MB cgroup) this already produced one OOM-kill on 2026-05-29 15:46, and will recur.

Code 
market_state.py:442-460   # _mark_settled: dataclasses.replace(q, settled=True) — kept forever
market_state.py:281-284   # only _evict_old_trades prunes; questions never pruned
scanner.py:212            # for q in self.ms.all_questions(): — iterates settled ones too
Confirmed: grep for any deletion/prune of _questions across engine/ returns nothing.
Proposed direction
  • Evict questions a bounded time after settlement (e.g. drop once settled & no open position & past a short grace window), or cap _questions with LRU on settled entries.
  • Have the scanner skip settled questions explicitly so eviction lag never re-trades them.
  • Optionally prune/rotate the seen_question table.
Acceptance
  • RSS stays bounded over a multi-day run (no monotonic growth in questions count).
  • No settled question re-enters the scan loop after eviction.

TE-4 · No per-market circuit breaker on consecutive order rejects

Medium area:engine reliability improvement
Problem

The router logs/publishes a reject and moves on; the only cooldown is a post-exit entry cooldown, nothing keyed on rejects. hl_client retries only on ConnectionError. So a market that keeps rejecting (post-only-style "could not immediately match") gets hammered every scan with no backoff.

Evidence
  • v31 #850 (q 1000085): 605 rejected orders, ~1 every 1.8 s for 87 min, zero fills until the book moved.
  • v31 overall reject rate 75% (655 / 877). Wasted API budget + rate-limit exposure.
Code 
router.py:180-221   # _place: logs OrderRejected, no breaker/backoff
hl_client.py:202-207 # @retry only on ConnectionError (not venue rejects)
Proposed direction

After N rejects on the same (question_idx, side) within T seconds, apply a short per-market backoff before re-quoting. Emit a metric/alert when the breaker trips.

Acceptance
  • A repeatedly-rejecting market is capped at ≤ N attempts per window instead of unbounded.

TE-5 · Telegram alert delivery failing

Low observability area:engine
Problem

Three consecutive Telegram send attempts failed around 2026-05-31 00:00 (telegram error attempt=1/2/3), so the reconcile-drift / reject-burst alerts of that window were never delivered — an operator blind spot, no trading impact.

Proposed direction

Add retry-with-backoff + a fallback delivery path (or a heartbeat "alerts healthy" check), and log undelivered alert bodies locally so nothing is lost.

Acceptance
  • Alerts survive a transient Telegram failure (delivered on retry or persisted for replay).

TE-6 · Confirm 6 early settlements that predate log/parquet retention

Medium investigation data
Problem

Six early held-to-settlement markets (q 12/13/14, 1000065/1000070/1000075) settled before journald/parquet retention, so win/loss is unconfirmed. They were bought at 0.87–0.999, so the asymmetry is stark: ≤ +$57 combined if all won vs ≈ −$590 combined if all lost. Given the 23/23 confirmed rate they almost certainly won, but it is unproven — this is the single largest residual PnL uncertainty in the eval.

Proposed direction

Pull HL userFills / account fill history for the account(s) directly (recorder retention doesn't reach these dates) and reconcile the 6 outcomes. Fold the result into the true-PnL number.

Acceptance
  • Win/loss known for all 6 markets; true-PnL updated and the uncertainty band closed.
