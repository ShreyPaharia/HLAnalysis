One strategy brain, two bodies.

A market-making research & trading stack for Hyperliquid binaries and Polymarket daily markets. The live engine and the backtester are deliberately built around a single shared spine of strategy + market-data code — so a tuned parameter set behaves the same in simulation and in production.

01  The four subsystems

Recorder → data → engine & backtester

Everything is one package, hlanalysis, exposed through four CLI entry points. The recorder captures reality to disk; the backtester replays that disk to tune parameters; the engine runs the tuned parameters live. The strategy code in the middle is the same object in all paths.

Recorder (capture) Shared core Live engine Backtester / sim

recorder  hl-recorder

Long-running collector. Subscribes to HL + Polymarket book/trade/settlement streams and a Binance spot reference feed, normalizes everything to one event taxonomy, and persists partitioned Parquet/DuckDB on EC2 (then S3). PM L2 books are not backfillable — the recorder is the only source of truth.

recorder/{runner,writer,read,main}.py · 417 LOC

backtester  hl-bt

Replays recorded data through the exact strategy code the engine runs, scores PnL with a realistic fill+fee sim (hftbacktest), and runs walk-forward parallel grid sweeps to produce the params shipped in config/strategy.yaml.

backtest/ + backtest/{core,data,runner}/ · 11k LOC

engine  hl-engine

The trading runtime. Runs one strategy "slot" per market track, evaluates entry/exit each tick against a live reference feed, routes orders (paper or live) through a risk gate, and reconciles against venue truth. Co-located with the recorder on one EC2 box, isolated by systemd cgroups.

engine/ · 30 modules · 9.5k LOC

strategy + marketdata  shared

The decision brain. strategy/ holds the registry and every @register'd strategy; marketdata/ holds the OHLC bucketer, position math, and MarketState core. Both the engine and the backtester import and call this same code.

strategy/ 4k LOC · marketdata/ ~1k LOC

03  Shared vs Independent — the spine

04  Live engine — internals

Per-slot isolation around a shared feed

Each (strategy, account) pair is one AccountSlot with its own DAL, exec client, risk gate, router, scanner, and strategy instance. Only the WebSocket feed and the MarketState are shared across slots. runtime.py drives five concurrent async loops:

_ingest_loop

Single shared WS subscription → applies events to MarketState; reconnect backoff; fires a market_dirty wake on price moves; latches all slots halted after repeated feed failure.

_scan_loop

Per-slot, event-driven. Wakes on market_dirty within a min-interval, coalesces bursts, floors at an idle max-interval. Calls scanner.scan() → strategy.evaluate() → router.

_reconcile_loop

Periodic three-way merge of local DB vs venue open-orders vs clearinghouse. Adopts/alerts on drift, detects settlement, evicts settled questions. HL applies venue truth; PM is alert-only after startup.

_continuous_checks_loop

Daily-loss cap (read from venue truth, not DB), kill-switch file polling, stale-data halt (gated on global feed silence), PM unconfirmed-order & redemption watchdogs.

_heartbeat_loop / _stop_loss_loop

Liveness heartbeat + RSS self-halt before OOM on the 1 GB box; dedicated event-driven stop-loss enforcement that bypasses the risk gate for defensive closes.

Tick → order path: WS → _ingest_loop → MarketState.apply → market_dirty → Scanner.scan → strategy.evaluate → Router.handle → RiskGate.check_pre_trade → exec_client.place → StateDAL.book_fill → TradeJournal → EventBus → {Telegram, events_sink}

Venue-neutral exec

exec_client.py defines an ExecutionClient protocol; hl_client and pm_client implement it so router + reconciler are venue-agnostic. HL settlement arrives as fills; PM settlement is an on-chain redeem tracked separately.

Persistence

state.py (SQLModel/SQLite, Alembic-migrated): positions, orders, fills, events, trade journal. CachedStateDAL prefers venue fills so local realized-PnL equals venue truth by construction.

replay.py — the bridge

ReplayRunner deterministically replays recorded NormalizedEvents through the same MarketState + strategy.evaluate as live — logging-only, no routing. The engine-side path that meets the sim.

05  Backtester / sims — internals

Recorded data in, tuned parameters out

The backtester replays recorded inputs (the Binance spot BBO feed and recorded PM/HL L2 books at the live sampling cadence) — never synthetic klines, because that is what the engine actually consumes live. The per-question loop builds numpy event arrays, drives an hftbacktest exchange, and calls the shared strategy each scan step.

run_one_question — the sim loop

Collects per-leg book/trade + reference + settlement events → builds event_dtype arrays → one HashMapMarketDepthBacktest asset per leg. Each hbt.elapse(scan_interval) drains reference events into MarketState, builds QuestionView + BookState, and calls strategy.evaluate(...). IOC orders fill against real recorded depth; open positions settle at the resolved outcome.

Parallel walk-forward tuning

tuning.py takes the Cartesian product of a grid YAML; each (cell, split) is independent work over a ProcessPoolExecutor (spawn). Workers rebuild the data source from a picklable SourceConfig — not env side-channels — and pin the parent checkout to sys.path[0] before any import. Results stream atomically to results.jsonl (resumable).

Realistic execution model

Partial fills against recorded book depth · constant/interpolated order latency (≈50 ms HL median) · slippage haircut · IOC marketability re-check to veto fleeting quotes. Two fee models: flat (HL, fee≈0) and pm_binary (fee = qty·rate·p·(1−p)).

Event-array cache

Per-question numpy bundles memoized to column-split .npz (~5.7× compression), keyed on sha256(question + files + build_version + config_sig). The config_sig captures reference resample dt + source + book mode, so a cadence change correctly invalidates. --no-cache / --rebuild-cache escape hatches.

Fidelity guardrails learned the hard way: recorded PM strike klines silently freeze (not error) when a cache is stale, and worker factories once dropped config-derived ctor params back to defaults via env-only propagation. Both are now fail-loud (StrikeCoverageError) or routed through the picklable SourceConfig. halt_replay.py additionally replays the engine's halt windows + inventory caps so the sim sees the same gating the live desk did.

07  Parity & strategy identity

How "the same code" is proven, not just claimed

Sharing code is only useful if you can show the two harnesses actually behave the same. Three mechanisms keep the engine and the backtester honest:

Identical strategy object

Both paths build the strategy through the registry from explicit params; the backtester's --slot mode reuses engine.config_builders directly. test_slot_config_parity asserts byte-for-byte config reproduction.

Shared MarketState

The engine's MarketState is a thin adapter over the same marketdata.MarketState the sim uses (SHR-87). σ cadence and reference source (mark/bbo, raw-tick vs bars) are resolved by the shared decision_input for both.

parity/ reconciliation

The parity package joins the engine's durable trade journal (SHR-83) against the sim's RunResult and attributes any residual into INPUT_SKEW, EXECUTION, or UNMODELED_HALT — turning "do they match?" into a measured number.

Why this architecture: a backtest is only trustworthy if the simulated decision is the decision the engine would have made. By collapsing OHLC, position math, MarketState, and the strategy itself into one shared spine — and proving equivalence with parity tooling — the team can tune on recorded data and deploy with confidence that the only thing that changed between sim and live is the source of the ticks.
