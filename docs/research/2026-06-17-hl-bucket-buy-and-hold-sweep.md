# HL v31 priceBucket — baseline (C3) vs buy-and-hold (`hold_msd2`) sweep

**Date:** 2026-06-17 · **Track:** HL HIP-4 daily BTC `priceBucket` ·
**Corpus:** 42 recorded questions, expiry 2026-05-08 .. 2026-06-18 (06:00 UTC) ·
**Engine:** `hl-bt run --slot v31 --slot-class priceBucket` (live-faithful slot
path), fee=flat $0, pruning on (decision-identical — see below).

This asks for `priceBucket` the same question the binary slot answered on
2026-06-17 (origin/main `41cdd28`): does switching from churn-prone mid-hold
exits to **buy-and-hold + a selective-entry `min_safety_d` gate** help?

## Configs

Both arms are the deployed slot config; only the `theta_overrides.priceBucket`
exit/entry knobs differ (everything else — favorite 0.85, edge_buffer 0.02,
vol_lookback 2700, dt 2, exit_spread_hold 0.04, tte from defaults 43200 — is
identical):

| knob | baseline (C3) | hold_msd2 |
|------|--------------|-----------|
| exit_safety_d | 1.0 | **0.0** (mid-hold exit off) |
| exit_edge_threshold | 0.0 | **1.0** (take-profit exit off) |
| min_safety_d | 0.0 | **2.0** (entry safety_d floor) |

`hold_msd2` is generated by `scripts/perf/make_bucket_hold_msd2_config.py` (exact
3-field patch of `config/strategy.yaml`, drift-guarded); the live config is NOT
touched. This is analysis only.

## Results (n=42)

| metric | baseline (C3) | hold_msd2 | Δ |
|--------|--------------:|----------:|---|
| total PnL | **$497.46** | **$560.01** | +$62.55 (+12.6%) |
| Sharpe (ann, /q) | **4.73** | 2.27 | −2.46 |
| hit rate | 69.0% | **76.2%** | +7.2pp |
| max drawdown | **$328.81** | **$582.79** | +$253.98 (+77%) |
| fills (entries / exits) | 587 (300 / 287) | 177 (143 / 34) | −70% |
| churn / question (entries / exits) | 7.14 / 6.83 | **3.40 / 0.81** | exits −88% |

**Churn:** buy-and-hold does what it did on binary — it collapses the churn.
Per-question exits drop from **6.83 → 0.81** (−88%) and total fills from 587 →
177 (−70%). The mid-hold/take-profit exit pair was driving the enter→exit→
re-enter cycling; turning it off (with `min_safety_d` gating the re-entries)
leaves mostly enter-once-and-hold.

**Tail (worst-5 per question):**

| baseline (C3) | | hold_msd2 | |
|---|---:|---|---:|
| q0035 (Q41) | −$148.35 | q0038 (Q56) | **−$500.00** |
| q0037 (Q55) | −$123.12 | q0020 (Q21) | **−$438.86** |
| q0020 (Q21) | −$79.70 | q0025 | $0.00 |
| q0034 (Q40) | −$71.59 | q0026 | $0.00 |
| q0028 (Q29) | −$26.53 | q0027 | $0.00 |

This is the whole story. `hold_msd2` makes slightly more money and hits more
often, but it does so by **holding through two favorites that went bad** —
q0038/Q56 −$500.00 (full max_position stake) and q0020/Q21 −$438.86 — where C3's
mid-hold `exit_safety_d` had cut the same positions long before the full loss
(**C3: Q56 −$8.92, Q21 −$79.70** → `hold_msd2`: **−$500.00, −$438.86**). With no
exit, a far-from-strike favorite that surprises takes the **entire stake** —
exactly the residual risk flagged for the binary cutover. The result: PnL/hit
improve at the mean while **max drawdown nearly doubles ($329 → $583) and Sharpe
halves (4.73 → 2.27)**.

## ⚠️ CAVEAT — do NOT read this as a deploy signal

- The 2026-06-05 bucket overfit rollback assumed the bucket corpus had **~zero
  losing favorites** (hence a tail-blind maxDD~$0). **That is no longer true in
  this n=42 window** — it now spans through 2026-06-18 and *does* contain losing
  favorites (Q21 05-28, Q56 06-15). So `hold_msd2`'s maxDD is **not** tail-blind
  here; it is a *realized* tail, and it is bad ($583, +77% vs C3).
- The mean-vs-tail trade is unfavorable for a buy-and-hold cutover on buckets:
  +$63 PnL / +7pp hit is bought with +$254 maxDD and two full-stake blowups.
  This is the **opposite** of the binary result (where the binary corpus tail
  was light and hold+`min_safety_d=2` turned the one loser into a small win).
- `min_safety_d=2.0` did **not** prevent the blowups — Q21/Q56 were favorites
  far enough from their adverse boundary to pass the entry floor, then drifted
  through it with no exit to react. A loss-injection stress (resample
  settlements at the favorites' implied loss rate, per the C3 rollback
  methodology) is still the required gate before any deploy, and on this
  evidence it would likely reject `hold_msd2` for buckets.
- **Recommendation: keep the live bucket config on C3.** Buckets are not binaries
  — the cliff-payoff tail makes the protective mid-hold exit load-bearing here.

## Cadence note (fidelity caveat)

Requested cadence was fixed 1s. On this box the full 42-question run at 1s
projected to ~1.5–2 h: the dominant per-cell cost is hftbacktest replaying the
deep 20-level early-corpus book event streams (cadence-*independent*), with the
per-scan σ/returns recompute (vol_lookback 2700 / dt 2 ⇒ 1350-sample window
every scan) on top. A single mid-corpus question exceeded 7 min at 1s. The sweep
was therefore run at **fixed 10s** (`--inner-scan-mode fixed --inner-scan-interval
10`, added to `resumable_run.py`). Implications:

- The **comparison is internally consistent** — both arms ran at identical 10s
  cadence via the identical slot-resolution path, differing only in the three
  exit/entry knobs.
- `hold_msd2` PnL is cadence-insensitive (enter-once, hold to settle). C3 PnL is
  mildly cadence-sensitive (exits fire within ≤10s of the threshold vs ≤1s), and
  the **churn counts are a lower bound vs 1s** (a coarser fixed grid throttles
  re-fires to ≤1 action/scan). The qualitative finding (hold_msd2 crushes churn
  but worsens the tail) is robust to cadence; exact 1s churn counts would be
  somewhat higher for C3.
- A 1s run is available on demand — the sweep is resumable
  (`scripts/perf/resumable_run.py`, warm chunks, no-orphan).

## Leg-pruning speed-up (behavior-preserving)

The loader now prunes untradeable `priceBucket` legs before decoding/replaying
them (`_is_prunable_bucket_leg`): NO/odd legs (theta only ever enters even/YES
legs) and YES legs whose best bid **and** best ask never reach the favorite
threshold over the question window (`_mid <= max(best_bid, best_ask)`, so the
favorite gate can never fire). A pruned leg is emitted as an empty no-quote leg.

- **Avg legs loaded per question: 6.00 → 1.38 (−77%)** (8-leg = 4-bucket
  questions; the 4 NO legs always prune, plus the non-favorite YES buckets).
- **Decision-identical:** validated on 8 questions — per-tick `action`, the
  numeric decision fields (p_model/edge/σ/τ), fills, and realized PnL are
  **bit-identical** with pruning off vs on. The only diff is a benign HOLD
  *reason* relabel (`no_favorite`→`no_book`) on no-entry ticks where a pruned
  tail leg's book is simply absent. (`scripts/perf/validate_bucket_leg_prune.py`)
- **Wall-time per question: 24.34s → 17.91s (−26%)** measured at 60s cadence
  (decode-dominated; off=load 6 legs, on=load ~1.4). At the sweep's finer cadence
  the saving is larger — pruning also drops those legs from the per-tick
  book-build + `apply_l2` loop and from the deep-book event-stream replay (the
  dominant cost; see cadence note), which is why the whole sweep was run with
  pruning on.
- **Why this is safe to run the sweep with:** the pruned legs can never be
  entered at *any* scan time (the favorite gate `_mid >= threshold` can never
  fire on them), so decision-identity is cadence-independent — the 10s sweep with
  pruning equals a 10s sweep without it by the same invariant proven bit-identical
  at 60s.

## Data-schema fix (unblocks the recent corpus)

Some 2026-06-15/16 multicoin-expansion bucket legs (e.g. `#4060`) have empty-book
parquet files whose `bid_px`/`ask_px` arrays are NULL-typed; DuckDB inferred the
union schema from the first file and crashed casting a real `DOUBLE[]` to
`NULL[]`. Fixed with `union_by_name=true` on the fast-path column reads (promotes
to the widest type; bit-identical on clean legs). Q73 (which spans 06-15..06-17)
now loads.
